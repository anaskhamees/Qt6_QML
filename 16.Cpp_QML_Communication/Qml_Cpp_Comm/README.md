# Qt QML - C++ Communication

**overview of how Qt Quick (QML) and C++ work together in modern Qt 6 applications.**

Qt Quick is the declarative UI framework built on top of Qt. While QML handles the visual layer (frontend), C++ is responsible for business logic, data processing, and performance-critical code (backend). The **Qt Meta-Object System (moc)** acts as the invisible bridge that makes seamless two-way communication possible.

---

##  Qt Quick Architecture

Qt Quick follows a **layered architecture**:

- **QML (Frontend / UI Layer)**: Declarative language for describing *what* the UI should look like and how it should behave. Uses a tree of visual items (`Rectangle`, `Text`, `Button`, custom components, etc.).
- **C++ (Backend / Logic Layer)**: Imperative code for complex calculations, network calls, file I/O, database access, hardware integration, etc.
- **Qt Meta-Object System (MoC)**: The communication bridge. Every QObject-derived class exposes its properties, signals, and methods at runtime via meta-data generated by the `moc` compiler.

---------------

## 1.  Qt Meta-Object System 

The **Qt Meta-Object System** is the core communication and reflection mechanism in Qt. It enables:

- Signals & Slots
- Q_PROPERTY system
- QML <--> C++ communication
- Runtime type introspection

> #### Runtime Type Introspection vs Reflection
>
> #####  Runtime Type Introspection
> Runtime type introspection is the ability of a program to **inspect its own types and object structure at runtime**.
>
> \- Reads type information (class name, properties, methods)
> \- Does NOT modify behavior
> \- Used for understanding object structure
>
> In short: **“Look at objects at runtime”**
>
> --------------------------------------------------------------------
>
> ##### Reflection
>
> Reflection is a broader concept where a program can **inspect and also modify or interact with its own structure at runtime**.
>
> \- Includes introspection (reading type info)
> \- Can call methods dynamically
> \- Can modify properties or behavior
> \- Enables dynamic execution
>
>  In short: **“Look at and control objects at runtime”**
>
> ###### Key Difference
> \- Introspection = Read only (inspect)
> \- Reflection = Read + Modify (inspect + act)

## 1.1. Meta-Object Compiler (moc)

`moc` is a Qt build tool that:

- Parses C++ header files
- Detects Qt macros
- Generates additional C++ code automatically
- It processes:

  - `Q_OBJECT`: Enables Qt Meta-Object System for the class
      
  - `signals`: Declares signals for event-based communication
      
  - `slots`: Declares callable functions in signal-slot system
      
  - `Q_PROPERTY`: Exposes properties to Qt Meta-Object System and QML
      
  - `Q_INVOKABLE`: Marks methods as callable from QML or meta-object system
      
  - `Q_ENUM` / `Q_ENUMS`: Exposes enums to Qt Meta-Object System and QML
      
  - `Q_FLAG` / `Q_FLAGS`: Exposes flag types (bitwise enums) to meta-system
      
  - `Q_CLASSINFO`: Attaches metadata (key-value info) to a class
      
  - `Q_NAMESPACE`: Allows exposing namespaces to the meta-object system (Qt 5.10+)
      

It generates files like: `moc_MyClass.cpp`

## 2.  Qt QML Engine (`QQmlApplicationEngine`)

`QQmlApplicationEngine` is the heart of every Qt Quick application. It is the runtime system of Qt.

```cpp
// main.cpp
#include <QGuiApplication>
#include <QQmlApplicationEngine>

int main(int argc, char *argv[]) {
    QGuiApplication app(argc, argv);
    
    QQmlApplicationEngine engine;
    engine.load(QUrl("qrc:/main.qml"));   // Loads QML and starts everything
    
    return app.exec();
}
```

### What the engine actually does:

1. **Loads QML files** (.qml, .js, resources).
2. **Creates the object tree** — every QML item becomes a QObject in memory.
3. **Handles property bindings** — continuously evaluates expressions.
4. **Manages object lifecycle** — automatic garbage collection via JavaScript engine + Qt ownership rules.
5. **Connects C++ with QML** — registers types, creates contexts, and routes signals/properties.

## 3.  QML Context 

Every QML file runs inside a **QQmlContext**.

- **Root Context**: The top-level context created by the engine (engine.rootContext()).
- Contexts form a **parent-child hierarchy** (like scopes in JavaScript).
- Properties and objects set on a context are **visible to all QML files** loaded in that context and its children.
- Anything exposed via setContextProperty or registered types becomes **globally visible**.
- Local contexts (created by Component, Loader, etc.) can have their own properties but inherit from parents.
- This is why you can access backend from any .qml file without importing.

------

### setContextProperty() 

```c++
// main.cpp
MyBackend backend;
engine.rootContext()->setContextProperty("backend", &backend);
```

Now in any QML file:

```js
Button {
    text: "Click me"
    onClicked: backend.doSomething()   // Direct access
}
```

## 4.  QML to C++ Communication (Frontend → Backend)

QML calls into C++ using three main mechanisms:

### 1. Q_INVOKABLE (Recommended modern way)

```c++
// MyBackend.h
class MyBackend : public QObject {
    Q_OBJECT
public:
    Q_INVOKABLE void processData(const QString& data);
};
```

```js
// QML
backend.processData("hello from QML")
```

### 2. slots (Legacy)

Same as Q_INVOKABLE but declared with public slots

```c++
public slots:
    void sayHello()
    {
        qDebug() << "Hello from C++ slot!";
    }
```

```js
   Button {
            text: "Say Hello"
            onClicked: backend.sayHello()
        }
```

### 3. Q_PROPERTY 

`Q_PROPERTY` is Qt’s way of turning a normal C++ variable into a **reactive, observable property** that can be used by:

- QML (UI layer)
- Qt Meta-Object System
- Signal/slot mechanism

It is the **bridge between C++ state and QML UI behavior**.

```
C++ variable
   ↓
Q_PROPERTY wraps it
   ↓
Becomes visible to Qt Meta-Object System
   ↓
QML can read / write it
   ↓
Changes automatically notify UI
```

####  Why it exists

Without `Q_PROPERTY`:

- QML cannot “see” C++ data
- UI cannot react automatically to changes

With `Q_PROPERTY`:

- C++ exposes state safely
- QML can bind to it
- UI updates automatically when data changes

---------------------------

####  What it actually provides

#### 1.  Read access (Getter) : QML can read the value:

#### 2.  Write access (Setter): QML can change the value:

#### 3.  Change notification (Signal): When value changes in C++:

- a signal is emitted
- QML UI automatically updates

-----------------------------------------------

#### Example 

------------------------------

```c++
Q_PROPERTY(int value READ value WRITE setValue NOTIFY valueChanged FINAL)
/*
  Exposes "value" to Qt Meta-Object System and makes it accessible from QML.

  - int value
      → Property type is int
      → Property name is "value" (used in QML as backend.value)

  - READ value
      → Getter function
      → Used by QML to read the property value
      → Must return the property type (int value() const)

  - WRITE setValue
      → Setter function
      → Called when QML assigns a new value (backend.value = 10)
      → Responsible for updating internal data

  - NOTIFY valueChanged
      → Signal emitted when the property changes
      → Triggers QML bindings to update automatically
      → Must be emitted inside the setter when value actually changes

  - FINAL
      → Prevents this property from being overridden in derived classes
      → Ensures property behavior remains fixed in inheritance chain
*/
```

`Q_PROPERTY` enables **reactive programming**:

-  You do NOT manually update UI
-  You only change data in C++
-  UI updates itself automatically

```
C++ changes value
   ↓
emit valueChanged()
   ↓
QML binding re-evaluates
   ↓
UI updates instantly
```

## 5.  C++ to QML Communication (Backend → Frontend)

This is where Qt shines — **reactive updates**.

### 1. Signals (emit)

```c++
signals:
    void dataChanged(const QString& newData);
```

In QML:

```js
Connections {
    target: backend
    function onDataChanged(newData) { console.log(newData) }
}
```

### 2. Q_PROPERTY

```c++
Q_PROPERTY(int counter READ counter NOTIFY counterChanged)

int m_counter = 0;

int counter() const { return m_counter; }

signals:
    void counterChanged();

public slots:
    void increment() {
        m_counter++;
        emit counterChanged();   // Triggers ALL bindings
    }
```

```js
Text { text: "Count: " + backend.counter }   // Auto-updates
```

**Note**: Backend never knows about the UI. It just notifies changes. The QML engine handles the rest.

## 6.  QML Property Binding System

This is the **magic** of Qt Quick.

```js
Text {
    text: "Value: " + backend.counter   // Binding expression
}
```

- Bindings are **reactive** and **lazy**.
- Whenever a NOTIFY signal fires, the binding engine re-evaluates the expression.
- Bindings can be nested, depend on other bindings, and even contain JavaScript.
- Difference from signal-slot:
  - Signal-slot = explicit connection.
  - Binding = **declarative dependency tracking** (automatic, bidirectional when possible).

The binding engine is part of the QML runtime and is extremely efficient.

## 7.  C++ Exposure Methods to QML

Qt provides multiple ways to expose C++ functionality to QML. Each approach has different use cases depending on scalability, architecture, and Qt version.

| Method                 | Use Case                               | Qt Version      | Advantages                                               | Disadvantages                                             |
| ---------------------- | -------------------------------------- | --------------- | -------------------------------------------------------- | --------------------------------------------------------- |
| `setContextProperty()` | Quick exposure of global objects       | All Qt versions | Very simple and fast to use                              | Pollutes global QML context, weak architecture separation |
| `qmlRegisterType<>()`  | Traditional type registration          | Qt 5 / Qt 6     | Structured, reusable QML types                           | Requires URI/module setup, more boilerplate               |
| `QML_ELEMENT`          | Modern QML type exposure (recommended) | Qt 6.2+         | Clean, no manual registration, integrates with CMake     | Requires modern Qt build system (CMake + QML modules)     |
| `QML_SINGLETON`        | Global shared service objects          | Qt 6.2+         | Ideal for shared services (settings, logger, API client) | Must be used carefully to avoid tight coupling            |

------

###  Modern Qt 6 Example (`QML_ELEMENT`)

```cpp
// MyBackend.h
#pragma once

#include <QObject>
#include <QtQml/qqml.h>

class MyBackend : public QObject
{
    Q_OBJECT
    QML_ELEMENT

    // Optional: Use if you want a global instance
    QML_SINGLETON

    Q_PROPERTY(int counter READ counter NOTIFY counterChanged)

public:
    int counter() const { return m_counter; }

signals:
    void counterChanged();

private:
    int m_counter = 0;
};
```

#### QML_SINGLETON vs QML_ELEMENT (Key Architectural Difference)

Qt provides two major ways to expose C++ classes to QML, but they serve fundamentally different architectural roles.

####  1. QML_ELEMENT (Type-based approach)

`QML_ELEMENT` exposes a C++ class as a **QML type (blueprint)**.

- You can create multiple instances in QML
- Each instance has its own independent state
- Works like a "class definition"
-  Each object is separate
- No shared state automatically

##### Example:

```js
BackendClass {
    id: obj1
}

BackendClass {
    id: obj2
}
```

------------------------------------------

#### 2. QML_SINGLETON (Global shared object)

`QML_SINGLETON` exposes a C++ class as a **single shared instance globally accessible in QML**.

- Only ONE instance exists in the entire application
- Shared across all QML files
- Accessed directly without creating objects (No need to instantiate)

##### Example:

```
Text {
    text: BackendClass.counter
}

Button {
    onClicked: BackendClass.increment()
}
```

##### Differences

| Feature      | QML_ELEMENT           | QML_SINGLETON          |
| ------------ | --------------------- | ---------------------- |
| Instances    | Multiple              | One only               |
| State        | Per object            | Shared global state    |
| Access style | `MyType {}`           | `MyType.property`      |
| Use case     | UI components, models | App services, settings |
| Memory model | Instance-based        | Global lifetime        |

#### When to use each?

#### Use QML_ELEMENT when:

- You need multiple objects
- Example:
  - List items
  - Data models
  - UI components

#### Use QML_SINGLETON when:

- You need shared application state
- Example:
  - App settings
  - Logger
  - Authentication state
  - Network manager



------------------------

#### Example :

```c++
/*
========================================
 Qt QML_SINGLETON Execution Flow
========================================

STEP 1: moc generates meta-data
---------------------------------
Your class is registered in Qt's Meta-Object System:

- class name     = Counter
- properties     = counter
- methods        = increment()

This is done by the moc (Meta-Object Compiler) at build time.


STEP 2: QML module registration (CMake / QML system)
----------------------------------------------------
Because of QML_ELEMENT:

- Qt registers the class into the QML type system

So QML now knows this type exists:

    Counter


STEP 3: Singleton creation (critical step)
------------------------------------------
Because of QML_SINGLETON:

- Qt guarantees ONLY ONE instance of this class exists

Internally:

- Counter instance is created ONCE
- Stored inside the QML engine
- Managed globally by Qt runtime


STEP 4: QML engine exposes it globally
--------------------------------------
Qt automatically injects it into the QML namespace:

    Counter → singleton instance reference

So in QML: Counter.counter

means: globalSingletonInstance.counter


STEP 5: Key Insight
-------------------
- You are NOT creating an object in QML
- You are accessing a Qt-managed global singleton instance


STEP 6: Internal Qt flow (conceptual)
-------------------------------------
QML Engine startup
        ↓
Detect QML_SINGLETON type
        ↓
Create C++ instance (Counter)
        ↓
Store in internal registry
        ↓
Expose globally as "Counter"
        ↓
Available in all QML files


STEP 7: Why Counter.increment() works
-------------------------------------
Because:

    Q_INVOKABLE void increment();

Qt Meta-Object System exposes it as:

- callable method from QML

So when QML executes:

    Counter.increment()

Qt internally does:

- find singleton instance
- invoke method via meta-object system
- execute C++ function


STEP 8: Why no Counter {} is needed
-----------------------------------
Comparison:

- QML_ELEMENT only:
    → You must create instances manually in QML

- QML_SINGLETON:
    → Qt creates and provides the instance automatically


STEP 9: Mental model
--------------------
Without singleton:
    QML = "I create objects"

With singleton:
    QML = "I use a globally managed shared object"


STEP 10: Final concept
----------------------
- QML_SINGLETON tells Qt to create and register exactly one global instance
   of the class inside the QML engine, making it accessible everywhere as
   "Counter".
*/


class Counter : public QObject
{
    Q_OBJECT
    QML_ELEMENT
    QML_SINGLETON

    Q_PROPERTY(int counter READ counter WRITE setCounter NOTIFY counterChanged)

public:
    explicit Counter(QObject *parent = nullptr);
    int counter() const { return m_counter; }

    void setCounter(int value);
    Q_INVOKABLE void increment();
signals:
    void counterChanged();

private:
    int m_counter = 0;
};
```

## 8.  Communication Types Summary

| Direction | Mechanism           | Best For            | Qt 6 Recommendation |
| --------- | ------------------- | ------------------- | ------------------- |
| QML → C++ | Q_INVOKABLE         | Direct method calls | **Preferred**       |
| QML → C++ | slots               | Legacy code         | Avoid               |
| QML → C++ | Q_PROPERTY WRITE    | Setting values      | Good                |
| C++ → QML | Signals             | Events              | Good                |
| C++ → QML | Q_PROPERTY + NOTIFY | Reactive data       | **Best practice**   |
| QML → QML | Property bindings   | UI reactivity       | Heavily encouraged  |

------

## 9.  Best Practices (Qt 6+)

-  Prefer QML_ELEMENT and QML_SINGLETON
-  Use Q_PROPERTY + NOTIFY for all exposed data
-  Prefer bindings over manual Connections and onXXX handlers
-  Avoid overusing setContextProperty() — use proper modules instead
-  Keep C++ classes clean and focused (single responsibility)
-  Use QML_SINGLETON for global services (logger, settings, network manager)
-  Always emit NOTIFY signals when properties change
-  Use CMake with qt_add_qml_module() for modern projects

------

## 10.  Architecture Idea

```
┌─────────────────────────────────────┐
│           QML = UI Layer            │
│   (Declarative, reactive, visual)   │
├─────────────────────────────────────┤
│      Qt QML Engine + Contexts       │
│          (The Bridge + Runtime)     │
├─────────────────────────────────────┤
│        C++ = Business Logic         │
│   (Data processing, models, APIs)   │
└─────────────────────────────────────┘
```

**Clean separation of concerns**:

- QML owns **how it looks and feels**.
- C++ owns **what it does**.
- The Qt engine owns **how they talk**.
